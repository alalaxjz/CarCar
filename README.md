## 2. 專案簡介

### 2.1 專案概述

本專案的目標是製作一台可以在迷宮地圖中自動行進的循跡自走車。車子會利用紅外線感測器偵測地圖上的黑線，沿著路徑前進，並在節點根據事先規劃好的方向指令進行直走、左轉、右轉或迴轉。

地圖中部分死路節點放有 RFID 卡片，代表可得分的寶藏點。當車子行進到寶藏點時，車上的 MFRC522 RFID 模組會讀取卡片 UID，並透過 HM-10 藍牙模組將 UID 傳送到電腦端。電腦端 Python 程式會接收 UID，檢查格式後送至 scoreboard server，更新隊伍分數。

整體系統可以分成三個主要層次：

1. **車體控制**：Arduino Mega 2560 負責循跡、控制馬達、判斷節點、執行轉向指令、讀取 RFID 並透過 HM-10 傳送 UID。
2. **演算法**：C++ 程式讀取地圖 CSV，建立加權圖，使用 Dijkstra 與 bitmask DP 規劃尋寶路徑，最後輸出車端可使用的 f, r, l, b 指令字串。
3. **藍牙與計分**：Python 程式透過 ESP32 與 HM-10 連線，接收車端傳來的 UID，並將 UID 送至 scoreboard server 計分。

### 2.2 專案目標

- 讓車子能穩定沿黑線行進，並在偏離黑線時自動修正方向。
- 讓車子能在節點根據指令執行直走、左轉、右轉與迴轉。
- 讓車子能在死路寶藏點讀取 RFID UID。
- 讓車子能透過 HM-10 藍牙模組把 UID 傳送到電腦端。
- 讓電腦端能將 UID 傳給 scoreboard server，並取得本次得分、剩餘時間與目前總分。
- 透過演算法自動計算最佳尋寶路徑。
- 在不同光線與地面環境下調整循跡參數，提高車子行進穩定度。

### 3.2 架構說明

### 3.2.1 路徑規劃層

路徑規劃由 `traversal.cpp` 完成。程式讀取 `big_maze_114.csv`，每個節點包含四個方向的鄰居與距離：

index, North, South, West, East, ND, SD, WD, ED

其中：

- `index`：節點編號。
- `North / South / West / East`：該方向相鄰節點。
- `ND / SD / WD / ED`：往該方向移動的距離權重。

程式會把地圖建成 weighted graph，並將 degree 為 1 的節點視為死路。除起點外，其餘死路節點則視為寶藏目標點。程式接著計算目標點之間的最短距離與路徑，並規劃拜訪所有寶藏點的順序。

### 3.2.2 車體控制層

車體控制由 `v16.ino` 完成。Arduino Mega 2560 讀取五顆紅外線感測器，根據黑線位置控制馬達速度與方向。當感測器偵測到節點時，Arduino 會讀取指令字串 `s[]` 中的下一個字元，執行對應動作。

指令字元意義如下：

| 指令 | 意義 | 車端動作 |
| --- | --- | --- |
| `f` | forward | 通過路口並繼續前進 |
| `r` | right | 右轉 |
| `l` | left | 左轉 |
| `b` | back / dead-end turn | 迴轉；也作為觸發 RFID 讀取的關鍵狀態 |

### 3.2.3 通訊與計分層

車端使用 `Serial3.print(buffer)` 將 RFID UID 傳給 HM-10。電腦端透過 ESP32 與 HM-10 建立 bridge，再由 Python 程式讀取 ESP32 串口輸出。

Python 主程式 `main.py` 會進行以下工作：

1. 初始化 scoreboard。
2. 開啟指定 COM port。
3. 確認 HM-10 名稱是否為 `HM10_Blue`。
4. 確認 ESP32 與 HM-10 是否為 connected 狀態。
5. 背景監聽藍牙訊息。
6. 若收到 8 位 16 進位 UID，則呼叫 `process_uid()`。
7. `process_uid()` 呼叫 `sb.add_UID(uid)`，將 UID 傳給 scoreboard server。
8. 顯示得分、剩餘時間與目前總分。

## 4. 專案功能特點（Features）

本專案的功能特點集中在三個部分：**路口轉彎／迴轉設計、脫線修正、路徑規劃演算法**。以下只整理這三項重點，不再展開其他一般模組內容。

## 4.1 路口判斷與轉彎／迴轉設計

本專案以紅外線感測器、路徑指令字串與左右輪差速控制完成路口導航。車子行進時會持續讀取五顆紅外線感測器；當中間三顆感測器同時偵測到黑線時，系統判定車子進入路口，並讀取指令字串 `s[]` 中的當前指令，執行直走、左轉、右轉或迴轉。

| 指令 | 動作 | 說明 |
| --- | --- | --- |
| `f` | 直走 | 通過路口並繼續循跡 |
| `l` | 左轉 | 在路口轉入左側路線 |
| `r` | 右轉 | 在路口轉入右側路線 |
| `b` | 迴轉 | 用於死路寶藏點刷卡後掉頭離開 |

左右轉採用差速轉彎。轉彎時降低其中一側輪子的 PWM，另一側維持較高速度，使車體轉向。由於左右馬達輸出、車體重心與輪胎摩擦力並非完全對稱，左轉與右轉使用不同參數，並依實車測試結果調整。轉彎過程中，程式搭配外側紅外線感測器判斷車體是否離開原本黑線區域，再以短暫 delay 補足轉向角度，以提高轉彎穩定度。

迴轉主要用於死路寶藏點。車子進入死路後讀取 RFID，接著須掉頭回到原路徑。迴轉時左右輪反向轉動，使車體能在有限空間內原地或近似原地旋轉。程式透過紅外線感測器判斷黑線離開與重新偵測的時機，使車子在迴轉後重新回到循跡狀態。

## 4.2 倒退修正與脫線恢復設計

本專案設計倒退修正機制，以處理車子偏離黑線、轉彎後未對準路線，或路口判斷後車體位置不穩的情況。當五顆紅外線感測器皆低於設定門檻時，系統判定車子可能脫線；程式會先短暫確認狀態，若仍未偵測到黑線，則啟動倒退修正流程。

```
五顆感測器皆未偵測到黑線
        ↓
判定可能脫線
        ↓
車子短暫後退
        ↓
左右側感測器重新搜尋黑線
        ↓
依偵測方向進行左修正或右修正
        ↓
中央感測器重新對準黑線
        ↓
恢復正常循跡
```

倒退修正的判斷依據為左右側感測器的回線狀態。若左側感測器先重新偵測到黑線，代表黑線位於車體左側，車子執行左修正；若右側感測器先重新偵測到黑線，代表黑線位於車體右側，車子執行右修正。修正持續至中央感測器重新偵測到黑線為止。

部分路口動作前也會先執行短暫後退。車子偵測到路口時，車體可能已進入大片交叉黑線區域；若直接轉向，感測器可能持續觸發路口判斷，或使車體卡在交叉線上。短暫後退可使車體退回較適合轉向的位置，再執行直走、左轉、右轉或迴轉，提高路口動作的一致性。

## 4.3 演算法

本專案使用 C++ 程式將迷宮地圖轉換為 graph，自動計算尋寶路徑，並輸出 Arduino 車端可執行的指令字串。演算法流程分為地圖建模、最短路徑計算、拜訪順序最佳化與車端指令轉換。

```
讀取地圖資料
        ↓
建立 weighted graph
        ↓
找出寶藏目標節點
        ↓
計算目標點間最短路徑
        ↓
規劃尋寶順序
        ↓
轉換為車端指令
```

### 4.3.1 地圖建模

本專案使用 `big_maze_114.csv` 作為地圖資料。CSV 中每一列代表一個節點，並記錄該節點在北、南、西、東四個方向的相鄰節點，以及往各方向移動的距離。

```
index, North, South, West, East, ND, SD, WD, ED
```

| 欄位 | 意義 |
| --- | --- |
| `index` | 節點編號 |
| `North / South / West / East` | 四個方向的相鄰節點 |
| `ND / SD / WD / ED` | 往各方向移動的距離權重 |

程式將每一條可通行路段記錄為 edge：

```
structEdge {
char dir;
int target;
int weight;
};
```

| 成員 | 意義 |
| --- | --- |
| `dir` | 移動方向，包含 `N / S / W / E` |
| `target` | 相鄰節點編號 |
| `weight` | 該路段距離 |

CSV 讀取完成後，系統建立 weighted graph。degree 為 1 的節點被視為死路節點，也就是 RFID 寶藏目標點。`traversal.cpp` 中設定起點為 `25`，其餘死路節點作為 targets。

### 4.3.2 最短路徑計算：BFS 至 Dijkstra

早期 `BFS.cpp` 版本以 BFS 計算路徑，適用於每條邊距離相同的地圖。BFS 能找到通過邊數最少的路徑，但無法處理不同路段距離造成的加權差異。

後續 `traversal.cpp` 使用 `big_maze_114.csv` 中的距離欄位建立 weighted graph。由於不同路段的距離可能不同，最短路徑應以總距離為準，而非以經過節點數為準。因此程式改用 Dijkstra 演算法計算任兩個目標點之間的最短距離。

Dijkstra 的輸出包含：

| 輸出 | 意義 |
| --- | --- |
| `dist` | 起點 target 到終點 target 的最短距離 |
| `dirs` | 沿途絕對方向序列 |

程式對所有 targets 兩兩執行 Dijkstra，並將結果存入 `memo_dist`，作為後續拜訪順序最佳化的距離表。

### 4.3.3 順序最佳化：Bitmask DP

當寶藏點數量大於一個時，系統需要決定拜訪順序，使總移動距離較短。本專案使用 bitmask DP 處理此類似 TSP 的問題。

DP 狀態定義為：dp[mask][u]

| 變數 | 意義 |
| --- | --- |
| `mask` | 已拜訪 target 集合 |
| `u` | 目前所在 target |
| `dp[mask][u]` | 此狀態下的最短總距離 |

此方法可系統性比較不同拜訪順序，避免僅選擇當下最近節點而導致整體路徑不佳。

### 4.3.4 車端相對指令轉換

Dijkstra 輸出的路徑為地圖上的絕對方向，例如 `N / S / W / E`；Arduino 車端執行的則是相對於目前車頭方向的指令，例如直走、左轉、右轉與迴轉。因此程式需將絕對方向轉換為 `f / l / r / b`。

程式使用 `Heading` 記錄目前車頭方向：

```
enumclassHeading {
NORTH,
EAST,
SOUTH,
WEST
};
```

轉換時比較下一步方向與目前車頭方向：

```
diff = (next-current+4)%4
```

| `diff` | 指令 | 意義 |
| --- | --- | --- |
| 0 | `f` | 維持方向，直走 |
| 1 | `r` | 右轉 |
| 2 | `b` | 迴轉 |
| 3 | `l` | 左轉 |

每次產生指令後，程式同步更新目前車頭方向，使下一步轉換能以新的車頭方向為基準。

### 4.3.5 演算法輸出

`traversal.cpp` 會輸出三項結果：

| 輸出 | 說明 |
| --- | --- |
| 指令字串 | Arduino 車端可執行的 `f / l / r / b` 指令 |
| 總距離 | 規劃路徑的總加權距離 |
| 理論總得分 | 根據死路節點距離計算的理論分數 |

## 8. 安裝與執行說明

### 8.1 安裝步驟

### 8.1.1 車體與硬體組裝

1. 依車體組裝投影片準備上板、下板、馬達支架、模組支架、電池支架等雷切零件。
2. 將兩顆直流減速馬達固定於車體。
3. 將萬向輪固定於車體前方或指定位置。
4. 固定五顆紅外線循跡模組。
5. 固定 MFRC522 RFID 模組。
6. 固定 Arduino Mega 2560、TB6612、降壓模組、HM-10 與 ESP32 相關連接。
7. 焊接 XT60、公母接頭、開關與馬達線。
8. 使用三用電表確認電池正負極、降壓輸出與 GND 是否正確。
9. 確認所有模組 GND 共地。
10. 開機前請先檢查電源正負極，避免反接燒毀電路。

### 8.1.2 Arduino 程式上傳

1. 開啟 Arduino IDE。
2. 選擇開發板：Arduino Mega 2560。
3. 開啟 `v16.ino`。
4. 確認已安裝 `MFRC522` library。
5. 確認程式中的感測器與馬達腳位符合實際接線。
6. 將程式上傳至 Arduino Mega 2560。
7. 開啟 Serial Monitor，可用於觀察 debug 訊息。

### 8.1.3 C++ 路徑規劃

1. 將 `traversal.cpp` 與 `big_maze_114.csv` 放在同一資料夾。
2. 編譯：g++ -std=c++17 traversal.cpp -o traversal
3. 執行：./traversal
4. 複製輸出的執行指令集。
5. 將該字串放入 Arduino 程式的 `char s[]`。

### 8.1.4 Python 端設定

1. 確認 ESP32 連接電腦後的 port。
2. 修改 `main.py` 中的 port。
3. 確認 server 位址："http://carcar.ntuee.org/scoreboard"
4. 執行。

## 10. 遇到的挑戰、對應的解方以及未來改進方向

### 10.1 USB 拔除後藍牙訊號不穩

### 問題描述

當 USB 線拔除後，藍牙訊號變得不穩定；刷 RFID 時會伴隨大量雜訊與亂碼。此問題影響車子獨立供電時的 UID 傳輸穩定性。

### 排查過程

一開始先檢查並修正程式碼，但問題沒有改善。接著觀察到電壓輸出在滿電壓與零之間反覆跳動，因此推測問題可能不是單純程式錯誤，而是硬體端，例如開發板或 HM-10 異常。

### 解決方法

先更換 HM-10 與 ESP32，但未獲改善。最後更換開發板後，問題順利修正。而 ESP32連接問題則可以用拔除與重插解決。

### 10.2 不同環境造成循跡不穩

### 問題描述

不同測試環境的光線與地板反射不同，會影響紅外線感測器判斷。例如在教室與宿舍光線不同時，感測器讀值會變化；宿舍地板甚至可能被判定為黑色，使車子偏離原本路線。

### 解決方法

團隊引入 PID 或調整循跡參數，以降低環境差異對車子行走的影響。實際使用的是門檻值判斷與左右輪 PWM 補償，並透過多組速度比例與 delay 數值調整車子動作。

### 10.3 讀不到特定卡片的 UID

### 問題描述

測試城中發現有些 RFID 卡片即使車子經過，score board 也沒有加分，檢查終端發現有刷到 UID，但依舊沒有將分數加上去。

### 解決方法

由於有刷到，排除 RFID 出問題的可能性，檢查藍牙與 server 整合也沒發現問題，最終發現是讀到並印出的 UID 會漏掉正確 UID 的 0 ，因此修正檢察十六進位的程式碼，最終成功解決。

### 10.4 未來改進方向

1. **路徑指令自動傳送**：目前車端指令寫死在 Arduino 程式中，未來可讓 Python 將指令透過藍牙逐點傳給車子，提升調整彈性。
2. **更完整的 PID 循跡**：將目前左右修正邏輯擴充成連續誤差控制。
3. **錯誤恢復機制**：若車子脫線或漏刷 RFID，可設計回復路線或重新定位策略。
